Fix: blank threads + sends creating stray branches (load-race bug)

The conversation-load effect marked a conversation as "loaded" BEFORE the fetch
resolved. A transient fetch failure (server restart, mobile network blip) then
wiped messages to empty yet left it flagged loaded, so it never retried — the
thread showed blank, and the next send (empty path -> parentId null) created a new
root/branch, which a refresh then revealed.

- Mark loadedId only AFTER a successful fetch; dedupe concurrent loads with a
  loadingId ref; retry up to 3x on failure; never wipe content on failure.
- computePath: if tree links are broken but messages exist, fall back to a linear
  time-ordered view so a thread never renders blank.
